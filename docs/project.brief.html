01 — Overview

What We're Building

A hybrid learning portal and portfolio web app. It serves two audiences: you, as the admin, recording notes and projects as you progress through your DevOps course — and employers or peers, who can browse your work through a clean public-facing view.

The app also doubles as a DevOps project in itself: containerised with Docker, deployed via CI/CD, and served behind Nginx — giving you live infrastructure to talk about.

02 — Core Features

What It Does

  • ▸
    Notion-style editor — Block-based rich text with headings, code blocks, and callouts. Powered by TipTap.
  • ▸
    Project showcase — Cards with tech stack tags, GitHub links, live demo links, and full detail pages.
  • ▸
    Interactive roadmap — A visual node graph showing your learning path. Nodes update visually as you progress.
  • ▸
    Public portfolio view — A read-only, employer-facing version of the app. No login needed to browse.
  • ▸
    Admin dashboard — Protected behind a login. Create, edit, and update all content from here.
  • ▸
    Tagging and filtering — Organise notes and projects by topic for easy navigation.
03 — Roadmap Page

Visual Progress Tracker

Built with React Flow, the roadmap renders your learning topics and projects as an interactive node graph — similar to roadmap.sh. Nodes have dependency connections (e.g. "Linux Basics" before "Docker"), and each node links directly to its associated notes page.

Node appearance changes based on status:

Not Started — muted, grey outline
In Progress — blue, pulsing ring
Completed — green fill, checkmark

As admin, you toggle status from a side panel that opens on node click. Changes fire a PATCH request and update the UI in real time. The public view reflects your live progress at all times.

Next.js 15 conventions for this phase:

params is a Promise — Dynamic route params must be awaited in every page component: const { id } = await params. Type as params: Promise<{ id: string }>.

getServerSession requires authOptions — Always pass authOptions explicitly: getServerSession(authOptions) where authOptions is imported from @/app/api/auth/[...nextauth]/route. Calling it without arguments may return null unexpectedly.

Server components query DB directly — Server components import from lib/queries/ and call the database directly rather than fetching their own API routes. API routes are for browser/client consumers only.

Postgres error 22P02 — invalid UUID — Passing a non-UUID string to a UUID column throws before Postgres even attempts the query. Catch 22P02 in all [id] route handlers and return 404, same as a not-found row.
04
Security Hardening
Closing vulnerabilities found during and after the admin build — before any public traffic
  1. 1
    HighLogin rate limiting — Added a login_attempts Neon table tracking attempts per IP with a fixed 15-minute window, 5 attempts allowed. Implemented in the NextAuth authorize callback via lib/queries/loginAttempts.ts. Counter resets on successful login. IP extracted from x-forwarded-for then x-real-ip, falling back to undefined — requests with no detectable IP are allowed through rather than sharing a shared "unknown" bucket. Lockout error message includes exact minutes remaining and is decoded from URL-encoding in submitLogin.ts before display. See Architecture Decisions — Login Rate Limiting.
  2. 2
    HighMagic byte validation for file uploads — file.type is browser-provided and can be spoofed. Added lib/validateFileBytes.ts which reads the first bytes of the uploaded buffer and checks them against known signatures for JPEG (FF D8 FF), PNG (89 50 4E 47...), GIF (47 49 46 38), and WebP (RIFF....WEBP). Detected MIME is also cross-checked against the declared file.type — a mismatch returns 400. Buffer is read once and reused for both validation and the R2 PutObjectCommand. See Architecture Decisions — Magic Byte Validation.
  3. 3
    HighUpload size limit corrected to 4MB — Original 5MB limit was above Vercel's 4.5MB serverless body limit, meaning the route-level validation was unreachable for large files. Vercel would reject the request first with a plain-text error body, causing a JSON parse failure on the client. Limit corrected to 4MB — safely below Vercel's infrastructure limit. Error message updated to match. Client-side res.json() wrapped in try/catch so non-JSON rejection bodies degrade gracefully.
  4. 4
    MediumLazy database client initialisation — Original lib/db.ts threw at module load time if DATABASE_URL was missing, causing pnpm build to fail because Next.js imports all route modules during the build step. Refactored to a lazy Proxy — the client is only created on the first actual query, not on import. Build now succeeds with no real database connection. See Architecture Decisions — CI and Database Initialisation.
  5. 5
    MediumCI build no longer uses real DATABASE_URL — Since the build does not connect to the database (all routes are dynamic, data fetched at request time), the CI workflow now uses a syntactically valid placeholder connection string instead of the production secret. Real DATABASE_URL stays on Vercel where it is only needed at runtime.
  6. 6
    Low/admin redirect — /admin with no sub-path returned a 404. Added /admin to the proxy.ts matcher and a pathname check that redirects to /admin/dashboard. Unauthenticated users still hit the auth check first and are sent to /admin/login.
  7. 7
    LowUpload response parsing hardened — handleImageUpload in EditorToolbar.tsx previously called res.json() unconditionally and used a non-null assertion on parsed.url. Fixed: res.json() wrapped in try/catch with logging, parsed.url explicitly guarded before calling editor.setImage(), non-null assertion removed. Error messages from the API route are decoded and surfaced directly rather than replaced with a generic fallback.
  8. 8
    MediumProject URL scheme validation — github_url and live_url were stored without validation. If rendered as href on public pages, values like javascript:... or data:... could lead to XSS or unexpected behaviour. Implemented in lib/validateProjectUrl.ts — isAllowedProjectUrlScheme() accepts only http: and https:, rejecting empty, invalid, or over-length URLs. normalizeProjectUrl() trims and normalises the value before validation. Both POST and PATCH handlers in /api/projects and /api/projects/[id] run normalise-then-validate before storing — invalid schemes return 400 with a clear error. Scheme enforcement is at write time so only safe values ever reach the database or the public-facing rendered links.
Why a dedicated phase?

Security work accumulated as a coherent body of changes after Phase 3 — each item identified through code review or feedback, fixed, tested, and documented. Grouping them as a named phase makes the decision trail explicit and is a stronger portfolio signal than scattered security commits across other phases. It demonstrates the ability to recognise, reason about, and systematically close vulnerabilities — a core DevOps skill.

Slug Management

When a project or note is created, the slug is auto-generated from the title using slugify() with a Date.now() suffix to guarantee uniqueness. The question is what happens after creation when the title changes.

Option A — Auto-update slug with title

Simple to implement. But dangerous once content is published — renaming a project would silently break any bookmarked or linked URL. Rejected.

Option B — Editable slug, manually controlled ✓

Slug is pre-filled from the title at creation but owned by the author from that point forward. A ↺ from title button regenerates it on demand. A warning appears when editing the slug of a published item. This is the pattern used by Notion, Ghost, and most CMSes.

ⓘ Both EditorPageClient.tsx and ProjectEditClient.tsx include an editable slug field with the ↺ regenerate button and a published-state warning. Slug input sanitises on keypress — only a-z, 0-9, and - are permitted.

Server Component Data Fetching

Server components fetch data by calling query helpers in lib/queries/ directly — not by calling their own API routes. This avoids an unnecessary HTTP round-trip from the server back to itself.

Server components → lib/queries/

Pages like /admin/notes and /admin/editor/[id] call getAllPages() or getPageById() directly. One function call, no HTTP overhead.

Client components → API routes

Browser code like CreateNoteButton and TipTapEditor call fetch('/api/pages') — they have no direct database access and must go through the API.

ⓘ Both getAllPages() (admin, all records) and getAllPublishedPages() (public, published only) live in lib/queries/pages.ts. The API route GET /api/pages handles the same filtering logic for client-side and external consumers.

How Notes Are Saved

TipTap produces a JSON document — a structured tree of nodes representing the content (paragraphs, headings, code blocks, etc.). That JSON is what gets stored, not HTML or markdown. In Postgres, the pages table holds a content column of type jsonb. When the editor loads, TipTap receives the JSON back and reconstructs the document exactly. When the public note page renders, TipTap's generateHTML() converts the JSON to HTML server-side.

We use autosave only — no manual save button. The editor saves automatically after the user pauses typing. A save status indicator in the meta bar shows the current state.

Autosave — 1.5s debounce

TipTap's onUpdate callback fires on every change. A 1.5-second debounce means the save only triggers once the user pauses. Title changes debounce at 1 second. Lightweight and unobtrusive.

Publish toggle

A Publish button in the editor meta bar toggles published between true and false via a PATCH request. Turns green when published. Reverts on error.

Save state indicator

A text label in the editor meta bar shows the current state: Saving… (yellow), Saved (green), or Save failed (red). Hidden when idle.

Why No ORM

An ORM (Object Relational Mapper) like Prisma sits between your application code and the database, letting you write TypeScript instead of SQL. Rather than writing SELECT * FROM projects WHERE id = $1, you'd write prisma.project.findUnique({ where: { id } }). It also handles connection management and generates TypeScript types from your schema automatically.

We considered both approaches before deciding. Here's the honest trade-off:

The case for an ORM

Less boilerplate for common queries. Schema file doubles as documentation. TypeScript types generated automatically. Migrations are straightforward. Fewer things to configure upfront.

The case against

Adds an abstraction layer you don't fully control. Requires learning the ORM's own API on top of everything else. Hides what's actually hitting the database. Overkill for a solo project with a stable schema.

What we chose: Raw SQL via postgres.js — a lightweight, modern Postgres client for Node.js. SQL queries are written directly in the API route handlers, and schema changes are managed as plain .sql migration files in a /migrations folder.

  • ▸
    Better learning value — Writing raw SQL builds a more fundamental understanding of how data is queried and structured. This matters far more in a DevOps context than knowing how an ORM works.
  • ▸
    More transparent — Every query is exactly what it looks like. There's no generated client, no magic, no debugging why the ORM produced an unexpected query plan.
  • ▸
    Closer to production practice — Most production teams manage migrations as versioned SQL files (or use lightweight tools like Flyway or Liquibase). Starting that way builds the right habits.
  • ▸
    Stronger talking point — "I write raw SQL and manage schema migrations manually" is a more compelling thing to say to an employer than "I used an ORM."

Media Cleanup Strategy

The problem. When TipTap removes an image from a note, it updates the document JSON — the <img> node disappears from the content tree. Nothing automatically tells R2 or the media table that the file is no longer in use. Without a cleanup strategy, deleted images accumulate as orphaned files in R2 and orphaned rows in the database indefinitely.

Why it's harder than it sounds. The naive approach — detect when an image URL disappears from the document and immediately delete it from R2 — has a critical edge case. If the user removes an image and then hits undo, TipTap restores the image node, but the file is already gone from R2. The image renders as broken. There's also the future consideration of the same image being reused across multiple notes — immediate deletion would break other references.

A second gap: R2 upload succeeds but DB insert fails. Even with a cron job sweeping the media table, there is a subtler failure mode: if the R2 PutObjectCommand succeeds but the subsequent INSERT INTO media fails, the object lands in R2 with no database record at all. Since the cron job queries the media table to discover what to clean up, it cannot find objects that were never recorded there. Those objects accumulate silently and invisibly.

Option A — Delete on removal

Diff image URLs on every TipTap onUpdate, delete any that disappeared. Simple but breaks undo for images — if a user undoes an image removal, the file is already gone from R2. Rejected as primary strategy.

Option B — Cron job + compensating delete ✓

Two-part strategy. A compensating DeleteObjectCommand fires immediately in the API route if the DB insert fails — synchronous, self-contained, closes the R2/DB gap at the source. A scheduled cron job handles the broader orphan sweep. Scheduled for Phase 7.

Option C — Ignore it

R2's free tier is 10GB — orphaned files are essentially free at this scale. Accept the orphans and clean up manually. Viable short-term but not a professional solution.

Decision: Option B — two-part strategy.

  • ▸
    Compensating delete in the API route (Phase 4, done) — if the DB insert fails after a successful R2 upload, a DeleteObjectCommand is issued immediately to roll back the R2 object. Synchronous and self-contained. If the rollback itself also fails, the error is logged and the failure is caught by the cron job's second pass.
  • ▸
    Cron job Pass 1 — DB-led sweep (Phase 7) — queries the media table for rows whose URL does not appear in any note's content JSONB, then deletes both the R2 object and the database row. Catches images removed from notes where the upload was correctly recorded.
  • ▸
    Cron job Pass 2 — R2 inventory reconciliation (Phase 7) — lists all objects in the R2 bucket via ListObjectsV2Command, cross-references against the media table, and deletes any R2 objects with no matching DB row. Catches objects that were uploaded to R2 but whose DB insert failed and whose compensating delete also failed. Belt-and-braces safety net.

ⓘ Together the three layers ensure no orphaned objects accumulate in either the database or the bucket — closing both the user-removal gap and the R2/DB consistency gap.

Login Rate Limiting

The credentials login endpoint has no built-in protection against brute force attacks. Without rate limiting an attacker who knows the admin email can script unlimited password attempts against a single known account. A medium severity issue identified during security review.

Option A — Upstash Redis + @upstash/ratelimit

Industry-standard approach for Next.js on Vercel. Runs at the edge, highly accurate across distributed nodes. Requires a free Upstash account and two additional environment variables. Rejected to avoid a new external dependency.

Option B — Vercel KV

Vercel's managed Redis (built on Upstash under the hood). Works with the same @upstash/ratelimit package. Fewer accounts to manage but still adds an external paid service. Rejected for the same reason as Option A.

Option C — In-memory Map in middleware

No external service — track attempts in a Map in the middleware module. Rejected because Edge middleware runs across Vercel's distributed network — each node has isolated memory, so an attacker hitting different nodes gets a fresh counter each time.

Option D — Neon DB tracking ✓

Track attempts in a login_attempts table using the existing Neon database. No new services, no new accounts. Implemented in the authorize callback in the NextAuth credentials provider via lib/queries/loginAttempts.ts.

Decision: Option D — Neon DB tracking. Uses the existing database, no new services required. The slight latency cost of an extra DB query on login is negligible for a route that fires rarely.

  • ▸
    Window — 5 attempts per IP per 15-minute fixed window. The window starts on the first attempt and resets after 15 minutes. A sliding window would close the burst-at-boundary edge case but adds significant complexity — unnecessary for a single-admin portfolio protected by bcrypt.
  • ▸
    Lockout message — Error includes exact minutes remaining: "Too many login attempts. Please try again in 14 minutes." Decoded from URL-encoding in submitLogin.ts before display.
  • ▸
    Reset on success — Counter is cleared via clearRateLimit(ip) after a successful login so a previously locked-out user starts fresh.
  • ▸
    IP extraction — Uses x-forwarded-for header set by Vercel in production. Falls back to x-real-ip for other reverse proxy setups, then undefined if neither header is present. See below for why 'unknown' was rejected as a fallback.
  • ▸
    Error passthrough — submitLogin.ts decodes and passes through the exact error message from authorize rather than hardcoding a generic fallback, so the rate limit message reaches the UI intact.

The "unknown" bucket problem. The initial implementation fell back to the literal string "unknown" when no IP could be determined:

const ip =
  (Array.isArray(forwarded) ? forwarded[0] : forwarded?.split(",")[0]) ?? "unknown"

The problem is that "unknown" is a shared key in the login_attempts table. If five failed attempts arrive from five different unidentifiable sources, they all increment the same counter and lock each other out. A legitimate user with no detectable IP would be locked out by an attacker's attempts — or vice versa. The bucket is shared across all requests that cannot be identified, making it both ineffective as a security control and a source of false positives.

Original — Fall back to "unknown"

All unidentifiable IPs share one rate limit bucket. Five failed attempts from different anonymous sources lock out all anonymous sources. Ineffective against a distributed attacker and causes false positives for legitimate users. Rejected.

Fix — Fall back to undefined, allow through ✓

If IP cannot be determined after checking x-forwarded-for and x-real-ip, pass undefined to checkRateLimit. The function returns { allowed: true } immediately without touching the database. An unidentifiable request bypasses rate limiting rather than sharing a poisoned bucket.

The trade-off is explicit — a request with no detectable IP is allowed through unconditionally. In practice on Vercel, x-forwarded-for is always populated in production so this path is only reached in unusual proxy configurations or local development (::1 normalised to 127.0.0.1).

ⓘ Migration: migrations/002_login_attempts.sql. Query helper: lib/queries/loginAttempts.ts — exports checkRateLimit(ip: string | undefined) and clearRateLimit(ip: string | undefined). Both functions are no-ops when ip is undefined.

Cron Job Reporting

Background jobs that run silently are a liability — you need to know whether they ran, what they did, and whether anything went wrong. For the media cleanup cron job three layers of observability are planned, each serving a different purpose.

Layer 1 — cron_log table

Every run writes a row to the cron_log table in Neon. Persistent, queryable, and free. The result JSONB column holds a structured pass-by-pass report. The errors column holds an array of any error messages. Status is success, partial (completed with some failures), or failed.

Layer 2 — Vercel logs

The result JSON is also piped to console.log on every run. Vercel captures these automatically and they're searchable in the dashboard. Zero cost, zero setup — useful for quick ad-hoc inspection without opening a DB client.

Layer 3 — Sentry alerts

Sentry (planned Phase 7) is configured to capture an exception when status is partial or failed. No notification on clean runs — alert fatigue from successful jobs is counterproductive. Only fire when human attention is needed.

R2 Image Upload Flow

When a user clicks the ⌃ Image button in the TipTap toolbar, an image travels from the browser through the Next.js API layer to Cloudflare R2, and the resulting URL is inserted inline into the editor and recorded in the database. Here is a breakdown of each stage.

① Browser — EditorToolbar

User clicks ⌃ Image → hidden <input type="file"> opens. On file selection, handleImageUpload fires. A FormData object is built with the file and the current note's ID as linked_to. A POST /api/media request is sent. The button shows Uploading… and is disabled until the response returns.

② API Route — /api/media

Session check first — unauthenticated requests are rejected with 401. Then validation in order: file present, linked_to is null or a valid UUID, MIME type is JPEG/PNG/WebP/GIF, size is under 4MB, R2 environment variables are present. Any failure returns a 4xx before touching R2 or the database.

③ Cloudflare R2

A unique filename is generated server-side: {Date.now()}-{random}.ext — the client's original filename is never trusted. The file buffer is uploaded via PutObjectCommand using the AWS S3-compatible SDK. The object is stored under the uploads/ prefix in the bucket.

④ Database — media table

After a successful R2 upload, a row is inserted into the media table recording the filename, full public URL, file size in bytes, and the note's UUID as linked_to. This record is what the Phase 7 orphan cleanup cron job will query to find unreferenced files.

⑤ Browser — TipTap editor

The API returns { url, filename }. The single parsed response object is used for both error handling and success. On success, editor.chain().focus().setImage({ src: url }).run() inserts the image inline at the cursor position. The image is immediately visible, served from the R2 public URL.

⑥ Future — Orphan cleanup

If a user later removes the image from the note, the media row and R2 object become orphaned. A scheduled cron job in Phase 7 will query for media rows whose URL does not appear in any note's content JSONB and delete both the database row and the R2 object.

End-to-end flow:

sequenceDiagram participant B as Browser (EditorToolbar) participant A as POST /api/media participant R as Cloudflare R2 participant DB as Neon (media table) B->>A: FormData { file, linked_to } A->>A: Auth check (session) A->>A: Validate file + linked_to + R2 config A->>R: PutObjectCommand { key, buffer, ContentType } R-->>A: 200 OK A->>DB: INSERT INTO media (filename, url, size, linked_to) DB-->>A: Row inserted A-->>B: { url, filename } 201 B->>B: editor.setImage({ src: url })

ⓘ The file extension is extracted server-side from the original filename but the filename itself is never used — a fresh Date.now()-random.ext key is always generated. Trailing slashes on R2_PUBLIC_URL are stripped before building the final URL to prevent double-slash paths.
